## 2. RAG 핵심 개념

### Q: "RAG가 뭔가요?"

> Retrieval-Augmented Generation의 약자입니다.
> LLM이 답변을 생성할 때, 외부 데이터베이스에서 관련 문서를 먼저 검색(Retrieval)해서
> 컨텍스트로 제공(Augmented)한 뒤 답변을 생성(Generation)하는 방식입니다.
>
> LLM만 쓰면 학습 데이터에 없는 정보는 답변할 수 없거나 할루시네이션이 발생합니다.
> RAG는 최신 데이터나 도메인 특화 데이터를 실시간으로 주입할 수 있어서 이 문제를 해결합니다.

### Q: "Fine-tuning 대신 RAG를 선택한 이유는?"

> 세 가지 이유입니다.
> 1. **비용**: Fine-tuning은 GPU 비용이 크지만, RAG는 기존 LLM을 그대로 사용
> 2. **실시간 업데이트**: Fine-tuning은 새 데이터마다 재학습 필요. RAG는 벡터DB에 추가만 하면 됨
> 3. **출처 추적**: RAG는 어떤 문서를 근거로 답변했는지 보여줄 수 있음 (특허 분석에서 중요)

### Q: "할루시네이션을 어떻게 줄였나요?"

> 프롬프트 설계에서 세 가지 규칙을 적용했습니다.
> 1. "검색된 문서의 내용만을 근거로 답변하세요" → 지어내는 것 방지
> 2. "출원번호를 반드시 인용하세요" → 출처 강제
> 3. "관련 정보가 없으면 모른다고 답하세요" → 무리한 답변 방지
> 4. temperature를 0.3으로 낮춰서 창의적 답변보다 정확한 답변 우선

## 3. 임베딩

### Q: "임베딩이 뭔가요?"

> 텍스트를 숫자 벡터(배열)로 변환하는 것입니다.
> 예: "배터리 기술" → [0.12, -0.45, 0.78, ...] (384차원)
>
> 의미가 비슷한 텍스트는 벡터 공간에서 가까이 위치합니다.
> "배터리 수명"과 "에너지 밀도 향상"은 키워드는 다르지만
> 임베딩 벡터는 가깝기 때문에 의미 기반 검색이 가능합니다.

### Q: "임베딩 모델을 왜 all-MiniLM-L6-v2로 선택했나요?"

> 1. **무료**: 로컬에서 실행 가능, API 비용 없음
> 2. **경량**: 384차원으로 Pinecone 무료 티어에 적합
> 3. **성능**: 문장 유사도 태스크에서 검증된 모델
> 4. **교체 가능**: 프로덕션에서는 OpenAI embedding으로 config만 변경하면 전환 가능
>
> 비용 최적화와 확장성을 동시에 고려한 설계입니다.

## 4. 벡터DB (Pinecone)

### Q: "벡터DB가 일반 DB와 뭐가 다른가요?"

> 일반 DB(PostgreSQL 등)는 정확한 값 매칭(WHERE name = '삼성')으로 검색합니다.
> 벡터DB는 벡터 간 유사도(cosine similarity)로 검색합니다.
>
> 예: "배터리 수명 기술" 검색 시
> - 일반 DB: 제목에 "배터리 수명"이 정확히 있는 것만 찾음
> - 벡터DB: 의미적으로 비슷한 "에너지 밀도", "충방전 사이클" 관련 특허도 찾음

### Q: "왜 Pinecone을 선택했나요?"

> 1. **관리형 서비스**: 인프라 관리 없이 API만으로 사용
> 2. **무료 티어**: 포트폴리오/데모용으로 충분 (1 인덱스, 2GB)
> 3. **확장성**: 트래픽 증가 시 자동 스케일링
> 4. **이전 회사 경험**: 프로덕션에서 Pinecone을 사용하는 환경에서 일했기 때문에 친숙
>
> 대안으로 Chroma(로컬)나 Weaviate도 있지만,
> 관리형 서비스의 운영 편의성과 이전 경험을 고려해 Pinecone을 선택했습니다.

### Q: "코사인 유사도가 뭔가요?"

> 두 벡터 사이의 각도를 측정합니다.
> - 1에 가까울수록: 의미가 비슷함
> - 0에 가까울수록: 관련 없음
> - -1에 가까울수록: 반대 의미
>
> 텍스트 유사도에는 코사인이 가장 적합합니다.
> 유클리드 거리(L2)는 벡터 크기에 영향을 받지만,
> 코사인은 방향만 비교하기 때문에 텍스트 길이에 덜 민감합니다.

## 5. 청킹

### Q: "청크 사이즈를 500으로 한 이유는?"

> 특허 문서의 특성을 분석한 결과입니다.
> - 특허 초록 평균 길이: 200~600자
> - 500자면 대부분의 초록이 1~2청크로 분할됨
> - 너무 작으면(100자): 문맥이 끊겨서 검색 품질 저하
> - 너무 크면(2000자): 검색 정밀도 저하 (관련 없는 내용도 포함)
> - overlap 50으로 청크 경계에서 문맥 유실 방지

### Q: "다른 청크 사이즈로 실험해봤나요?"

> 300, 500, 1000으로 비교했습니다.
> - 300: 청크가 너무 많아져서 검색 시 노이즈 증가
> - 500: 특허 초록 단위로 적절히 분할, 검색 정확도 가장 높음
> - 1000: 한 청크에 여러 주제가 섞여서 정밀도 저하
>
> 도메인에 따라 최적값이 다르기 때문에 실험이 중요합니다.

## 6. LLM

### Q: "왜 Ollama/llama3를 사용했나요?"

> 비용 최적화를 위해서입니다.
> - 개발/데모 단계에서는 Ollama로 비용 $0
> - 프로덕션 전환 시에는 config의 LLM_PROVIDER만 바꾸면 OpenAI나 Azure OpenAI로 교체 가능
> - 코드 변경 없이 config만으로 LLM을 교체할 수 있도록 추상화 레이어를 설계했습니다
>
> 이렇게 하면 개발 비용은 절감하면서, 프로덕션 요구사항에 유연하게 대응할 수 있습니다.

## 7. 인프라/Docker

### Q: "Docker를 왜 사용했나요?"

> 1. **환경 일관성**: "내 컴퓨터에서는 되는데" 문제 방지
> 2. **배포 용이**: Docker 이미지만 올리면 어떤 클라우드에서든 실행 가능
> 3. **팀 온보딩**: `docker compose up` 한 줄이면 전체 환경 셋업 완료
> 4. **마이크로서비스 대비**: 백엔드/프론트엔드를 독립적으로 스케일링 가능

### Q: "CI/CD 파이프라인을 설명해주세요"

> GitHub Actions로 구성했습니다.
> push → lint → 타입 체크 → 테스트(pytest) → 빌드 → 배포
>
> 각 단계가 실패하면 배포가 중단되어 프로덕션 안정성을 보장합니다.
> 이건 MLOps의 기본 파이프라인이기도 합니다.

## 8. 아키텍처 설계

### Q: "전체 아키텍처를 설명해주세요"

> 크게 두 가지 흐름이 있습니다.
>
> **인제스트 흐름** (데이터 저장):
> KIPRIS API → 특허 데이터 수집 → LangChain Document 변환
> → RecursiveCharacterTextSplitter로 청킹 → HuggingFace로 임베딩
> → Pinecone에 벡터 저장
>
> **검색 흐름** (RAG):
> 사용자 질문 → 질문을 임베딩 → Pinecone에서 유사 문서 top-k 검색
> → 검색된 문서를 컨텍스트로 프롬프트에 주입 → LLM이 답변 생성
> → 답변 + 출처 특허 반환
>
> 프론트엔드는 Next.js로, FastAPI 백엔드와 REST API로 통신합니다.
> 전체를 Docker Compose로 컨테이너화하여 한 명령어로 실행 가능합니다.

## 10. CS 기초

### Q: "HTTP 메서드 GET과 POST의 차이?"

> GET: 데이터 조회 (URL에 파라미터 노출, 캐싱 가능)
> POST: 데이터 생성/처리 (body에 데이터 포함, 캐싱 불가)
> TechDocs에서 검색은 POST로 — body에 복잡한 쿼리를 담아야 하기 때문

### Q: "REST API란?"

> HTTP 메서드(GET/POST/PUT/DELETE)로 리소스를 조작하는 API 설계 규칙
> TechDocs API: POST /api/search (검색), POST /api/ingest (인제스트), GET /health

### Q: "비동기(async)를 왜 사용했나요?"

> KIPRIS API 호출, Pinecone 검색, LLM 답변 생성 모두 I/O 바운드 작업입니다.
> async를 사용하면 대기 시간에 다른 요청을 처리할 수 있어 동시 처리 성능이 향상됩니다.
> FastAPI는 기본적으로 async를 지원하기 때문에 자연스럽게 적용했습니다.

### Q: "Docker와 가상머신의 차이?"

> 가상머신: OS 전체를 가상화 (무겁고 느림)
> Docker: OS 커널을 공유하고 앱만 격리 (가볍고 빠름)
> 컨테이너는 이미지 기반이라 어디서든 동일하게 실행 가능
