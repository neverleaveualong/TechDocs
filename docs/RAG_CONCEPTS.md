# RAG 핵심 개념 정리 (면접용)

---

## 1. RAG (Retrieval-Augmented Generation)

### 한 줄 정의
LLM이 답변할 때 외부 데이터를 검색해서 컨텍스트로 제공하는 기술

### 왜 필요한가
```
LLM만 사용:
  "더존비즈온의 최신 특허 알려줘"
  → "모르겠습니다" 또는 거짓 정보 생성 (할루시네이션)

RAG 사용:
  질문 → 벡터DB에서 관련 특허 검색 → 검색 결과 + 질문을 LLM에 전달
  → "더존비즈온의 최신 특허 3건은 다음과 같습니다. 1. ..."
```

### RAG vs Fine-tuning

| | RAG | Fine-tuning |
|---|---|---|
| 데이터 업데이트 | 벡터DB에 추가만 하면 됨 (실시간) | 재학습 필요 (시간+비용) |
| 비용 | 낮음 (기존 LLM 사용) | 높음 (GPU 학습 비용) |
| 출처 추적 | 가능 (어떤 문서 참고했는지 표시) | 불가능 |
| 적합한 경우 | 최신 데이터, 도메인 특화 | 모델 행동 자체를 바꿀 때 |

## 2. 임베딩 (Embedding)

### 한 줄 정의
텍스트를 고정 크기의 숫자 배열(벡터)로 변환하는 것

### 예시
```
"배터리 기술" → [0.12, -0.45, 0.78, ..., 0.33]  (384개 숫자)
"에너지 밀도" → [0.14, -0.42, 0.76, ..., 0.31]  (비슷한 벡터!)
"요리 레시피" → [0.89, 0.12, -0.55, ..., -0.22] (완전 다른 벡터)
```

### 왜 벡터로 바꾸는가
- 텍스트끼리는 "의미가 비슷한지" 수학적으로 비교할 수 없음
- 벡터로 바꾸면 코사인 유사도로 비교 가능
- "배터리 기술"과 "에너지 밀도"는 키워드가 다르지만 벡터가 가까움 → 의미 검색 가능

### 임베딩 모델 종류

| 모델 | 차원 | 비용 | 특징 |
|---|---|---|---|
| all-MiniLM-L6-v2 | 384 | 무료 (로컬) | 가볍고 빠름, TechDocs에서 사용 |
| text-embedding-3-small | 1536 | $0.02/1M토큰 | OpenAI, 높은 정확도 |
| text-embedding-3-large | 3072 | $0.13/1M토큰 | 최고 정확도 |

## 3. 벡터 데이터베이스 (Vector DB)

### 한 줄 정의
벡터(임베딩)를 저장하고, 유사한 벡터를 빠르게 검색하는 특수 데이터베이스

### 일반 DB vs 벡터 DB

```
PostgreSQL (일반 DB):
  SELECT * FROM patents WHERE title LIKE '%배터리%'
  → 제목에 "배터리"가 있는 것만 찾음

Pinecone (벡터 DB):
  query_vector = embed("배터리 수명 늘리는 기술")
  → 의미적으로 비슷한 벡터를 찾음
  → "에너지 밀도 향상", "충방전 사이클 개선" 등도 나옴
```

### 주요 벡터 DB 비교

| DB | 유형 | 특징 |
|---|---|---|
| Pinecone | 관리형 (클라우드) | 운영 부담 제로, 무료 티어 |
| Chroma | 로컬 | 설치 간단, 소규모 적합 |
| Weaviate | 하이브리드 | 벡터 + 키워드 검색 내장 |
| Milvus | 오픈소스 | 대규모, 자체 운영 필요 |
| pgvector | PostgreSQL 확장 | 기존 PG에 추가 가능 |

## 4. 코사인 유사도 (Cosine Similarity)

### 한 줄 정의
두 벡터 사이의 각도로 유사도를 측정 (1이면 동일, 0이면 무관)

### 수식
```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)
= 두 벡터의 내적 / (각 벡터의 크기의 곱)
```

### 왜 코사인인가
- 유클리드 거리: 벡터 크기(텍스트 길이)에 영향 받음
- 코사인: 방향만 비교 → 텍스트 길이에 덜 민감
- 텍스트 유사도에는 코사인이 표준

### 예시
```
"배터리 기술" vs "에너지 밀도" → 유사도 0.85 (높음)
"배터리 기술" vs "요리 레시피" → 유사도 0.12 (낮음)
```

## 5. 청킹 (Chunking)

### 한 줄 정의
긴 문서를 검색에 적합한 크기로 쪼개는 것

### 왜 필요한가
```
특허 문서 전체 (5000자)를 하나의 벡터로 만들면:
  → 벡터가 문서 전체의 "평균 의미"를 담음
  → 특정 기술을 검색할 때 정밀도가 떨어짐

500자씩 쪼개면:
  → 각 청크가 구체적인 내용을 담음
  → 검색 정밀도 향상
```

### 청크 사이즈 트레이드오프

| 작은 청크 (100~300) | 큰 청크 (1000~2000) |
|---|---|
| 높은 정밀도 | 넓은 문맥 |
| 문맥 끊김 위험 | 노이즈 포함 위험 |
| 청크 수 많아짐 | 청크 수 적음 |

### TechDocs 전략
- 500자 + overlap 50
- 특허 초록 평균 200~600자 → 대부분 1~2청크
- RecursiveCharacterTextSplitter: 문단 → 문장 → 단어 순으로 분할

## 6. 하이브리드 검색

### 한 줄 정의
벡터 검색(의미) + 키워드 검색(BM25)을 결합하는 방식

### 왜 필요한가
```
벡터 검색만:
  "10-2024-0012345" (출원번호) 검색 → 의미가 없어서 엉뚱한 결과

키워드 검색만:
  "배터리 수명 늘리는 기술" → "에너지 밀도" 관련 특허 못 찾음

하이브리드:
  벡터 검색 결과 + BM25 키워드 검색 결과 → 합쳐서 순위 매김
  → 의미적 유사도 + 정확한 키워드 매칭 둘 다 커버
```

### BM25란
- 전통적인 키워드 기반 검색 알고리즘
- 단어 빈도(TF)와 문서 빈도(IDF)를 기반으로 관련도 계산
- Elasticsearch의 기본 검색 알고리즘

## 7. 프롬프트 엔지니어링

### 한 줄 정의
LLM에게 원하는 답변을 얻기 위해 입력(프롬프트)을 설계하는 기술

### TechDocs 프롬프트 설계 원칙

```
1. 역할 부여: "당신은 특허 문서 분석 전문가입니다"
2. 컨텍스트 제공: 검색된 특허 문서를 {context}로 주입
3. 제약 조건: "검색된 문서만 근거로", "출원번호 인용"
4. 출력 형식: "구조화하여 읽기 쉽게"
5. 안전장치: "없으면 모른다고 답하세요" (할루시네이션 방지)
```

### Temperature

```
0.0: 항상 같은 답변 (결정적)
0.3: 약간의 변화 (TechDocs — 정확성 우선)
0.7: 다양한 답변 (일반 대화)
1.0: 창의적 답변 (소설 등)
```

## 8. LangChain

### 한 줄 정의
LLM 애플리케이션을 쉽게 만들 수 있게 해주는 프레임워크

### 핵심 컴포넌트

```
Document:        텍스트 + 메타데이터
TextSplitter:    문서 청킹
Embeddings:      임베딩 모델 래퍼
VectorStore:     벡터DB 래퍼
LLM/ChatModel:   LLM 래퍼
Chain:           여러 컴포넌트를 연결한 파이프라인
Retriever:       검색 로직 추상화
```

### TechDocs에서 사용하는 체인

```python
RetrievalQA.from_chain_type(
    llm=ollama,              # LLM (답변 생성)
    retriever=pinecone,      # 벡터 검색
    chain_type="stuff",      # 문서 결합 방식
    return_source_documents=True,  # 출처 반환
)
```

**chain_type 종류:**
- stuff: 모든 문서를 하나로 합쳐서 LLM에 전달 (간단, 문서 적을 때)
- map_reduce: 각 문서를 개별 처리 후 합침 (문서 많을 때)
- refine: 순차적으로 답변 개선 (정교함 필요할 때)

## 9. 데이터 파이프라인 전체 흐름

```
[인제스트]
KIPRIS API 호출
  → XML 응답 파싱 (xmltodict)
  → Python dict로 변환
  → LangChain Document로 변환 (page_content + metadata)
  → RecursiveCharacterTextSplitter로 청킹 (500자, overlap 50)
  → HuggingFace로 임베딩 (384차원 벡터)
  → Pinecone에 upsert (벡터 + 메타데이터 저장)

[검색]
사용자 질문
  → HuggingFace로 질문 임베딩
  → Pinecone에서 cosine similarity top-k 검색
  → 검색된 Document들을 프롬프트 {context}에 주입
  → Ollama(llama3)로 답변 생성
  → 답변 + 출처 특허 목록 반환
```

## 10. 용어 사전 (면접에서 자연스럽게 쓰기)

| 용어 | 의미 |
|---|---|
| Embedding | 텍스트를 벡터로 변환 |
| Vector | 숫자 배열 [0.1, -0.3, ...] |
| Chunk | 문서를 쪼갠 조각 |
| Top-k | 유사도 상위 k개 결과 |
| Upsert | Insert + Update (있으면 갱신, 없으면 추가) |
| Retriever | 검색기 (질문 → 관련 문서 반환) |
| Context | LLM에 제공하는 참고 정보 |
| Hallucination | LLM이 사실이 아닌 정보를 생성하는 현상 |
| Temperature | LLM 답변의 무작위성 조절 (0~1) |
| Token | LLM이 텍스트를 처리하는 단위 (대략 한글 1글자 ≈ 2~3 토큰) |
| Dimension | 임베딩 벡터의 크기 (384, 1536 등) |
| Index | 벡터DB에서 데이터를 저장하는 단위 |
| Namespace | Pinecone에서 데이터를 논리적으로 분리하는 단위 |
| BM25 | 키워드 기반 검색 알고리즘 |
| Cosine Similarity | 벡터 간 유사도 측정 (0~1) |
