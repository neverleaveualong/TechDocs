# 자동 수집 Agent 품질 개선 회고

## 배경

TechDocs는 KIPRIS 특허 데이터를 Pinecone에 임베딩해 두고, 사용자의 자연어 질의에 맞는 특허 근거를 찾아 RAG 답변을 생성한다. 그런데 실제 테스트에서는 "검색 결과가 없는 경우"보다 더 어려운 문제가 있었다. 검색 결과는 존재하지만 사용자의 질의와 무관한 특허가 top-k로 반환되는 경우였다.

예를 들어 `수소 연료전지 드론의 배터리 열관리와 비행 안정성을 개선하는 기술`을 검색했을 때, 기존 벡터에 저장되어 있던 CCTV/영상 감시 특허가 반환됐다. 기존 로직은 `sources`가 비어 있을 때만 KIPRIS 자동 수집을 실행했기 때문에, 무관한 결과가 있어도 "데이터가 있다"고 판단하고 자동 수집을 건너뛰었다.

## 어려웠던 점

가장 어려웠던 점은 자동 수집 자체가 아니라, "언제 수집해야 하는지"와 "어떤 후보를 임베딩해야 하는지"를 판단하는 일이었다.

초기에는 `sources` 유무만 봤기 때문에 검색 품질이 낮은 결과를 구분하지 못했다. 이후 retrieval score와 질의 키워드 overlap을 기준으로 `search_quality` 레이어를 추가했지만, 그 다음에는 후보 선택 문제가 드러났다. KIPRIS에서 후보 특허가 일부 검색되더라도 rerank score가 cutoff를 넘지 못하면 저장하지 않았고, 반대로 cutoff를 낮추면 관련성이 낮은 특허까지 임베딩될 위험이 있었다.

즉 trade-off는 다음과 같았다.

- cutoff가 높으면 무관한 특허 저장은 줄지만, 실제로 도움이 될 수 있는 후보까지 버릴 수 있다.
- cutoff가 낮으면 자동 수집 성공률은 올라가지만, Pinecone에 노이즈가 쌓여 RAG 답변 품질이 흔들릴 수 있다.
- 복합 질의는 하나의 특허가 모든 조건을 만족하지 않을 수 있다. `수소 드론`, `배터리 열관리`, `비행 안정성`처럼 여러 기술 축으로 나뉘는 경우, 단일 최고점 후보만 보는 방식은 한계가 있다.

## 개선한 방향

자동 수집을 두 단계로 나눠 판단하도록 개선했다.

1. 검색 품질 판정
   - source count
   - retrieval score
   - 질의에서 추출한 기술 키워드와 source 텍스트의 overlap
   - 자동 수집 필요 여부와 reason

2. 자동 수집 후보 선택
   - embedding rerank score
   - 후보 특허가 커버하는 핵심 feature 수
   - 서로 다른 기술 축을 가진 후보를 우선 선택하는 diverse selection
   - coverage가 없는 후보는 제외
   - KIPRIS 검색은 월 호출 제한을 고려해 query plan의 핵심 검색어를 `word`로 먼저 조회하고, title/abstract는 남은 attempt의 fallback으로 사용

이렇게 바꾸면서 단순히 `score >= 0.56`인 후보만 저장하지 않고, score는 조금 낮더라도 핵심 feature를 2개 이상 커버하는 후보는 선택할 수 있게 했다. 예를 들어 하나의 특허가 전체 질의를 모두 만족하지 않더라도, `수소 드론` 축과 `배터리 열관리` 축의 후보를 함께 저장해 RAG가 더 정직하게 답변할 수 있는 근거 묶음을 만들 수 있다.

## 배운 점

RAG 품질은 LLM 답변 프롬프트만으로 해결되지 않는다. 검색 결과가 충분한지, 기존 벡터가 질의와 맞는지, 새로 수집한 후보를 임베딩해도 되는지까지 데이터 흐름 전체의 품질 판단이 필요하다.

특히 자동 수집 Agent는 "많이 수집하는 것"보다 "수집해도 되는 데이터를 고르는 것"이 더 중요했다. 잘못된 후보를 임베딩하면 이후 검색 결과 전체가 오염될 수 있기 때문이다.

면접에서 이 경험은 다음처럼 설명할 수 있다.

> 초기에는 RAG 검색 결과의 sources 유무만 보고 자동 수집 여부를 판단했습니다. 하지만 실제 테스트에서 기존 벡터가 질의와 무관해도 top-k로 반환되면 시스템이 데이터가 충분하다고 오판하는 문제가 있었습니다. 이를 해결하기 위해 검색 품질 판정 레이어를 추가했고, retrieval score와 키워드 overlap을 기준으로 자동 수집을 트리거하도록 개선했습니다. 이후에는 단일 rerank cutoff만으로 후보를 고르면 복합 질의에서 유효 후보까지 버려지는 문제가 있어, 핵심 feature coverage와 후보 다양성을 기준으로 자동 수집 Agent를 고도화했습니다.

## 다음 개선 과제

- 자동 수집 결과를 운영 로그에 저장해 cutoff와 coverage 기준을 실제 질의 기반으로 튜닝한다.
- `low_relevance` 상태에서도 후보별 탈락 사유를 UI에 노출해 디버깅 가능성을 높인다.
- LLM 답변 정책을 보강해 직접 일치 특허와 관련 기술 축 후보를 구분해서 설명하게 한다.
- KIPRIS 검색 query plan을 더 세분화해 `연료전지 드론`, `드론 배터리 열관리`, `드론 비행 안정성` 같은 조합 질의를 안정적으로 생성하고, `word` 검색으로 넓힌 후보를 coverage/rerank로 걸러낸다.
